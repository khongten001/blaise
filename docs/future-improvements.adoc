= Blaise Compiler — Future Improvements
:toc:
:toc-placement: preamble
:sectnums:

This document records optimisation opportunities, quality-of-life improvements,
and language enhancements that are worth pursuing but are deliberately deferred
from the current implementation phase. Each entry states what the improvement is,
why it matters, and any known implementation notes.

Items here are not committed to any release milestone. They exist so that good
ideas are not lost and so that the rationale is available when the time comes to
prioritise them.

== Code Generation

=== Int64 literal range detection in self-hosted binary

*What.* When a constant or literal value exceeds the 32-bit signed range
(`> 2147483647`), the codegen should emit `l copy <value>` (64-bit) rather
than `w copy <value>` (32-bit).  Currently the range check in
`EmitIntLiteralExpr` is `Value > 2147483647`, which uses a 32-bit comparison
in the self-hosted binary; an Int64 value whose lower 32 bits appear negative
is incorrectly classified as in-range.

*Why it matters.* The current workaround is that `MaxInt` is declared as a
32-bit `Integer` constant (`2147483647`) rather than a true 64-bit value.
This is correct for `Copy(S, N, MaxInt)` usage (any string shorter than 2 GB
qualifies), but it means `High(Int64)` cannot be represented as a named
constant, and large Int64 literals in user code may be silently truncated.

*Root cause.* `TSymbol.ConstValue` is declared as `Integer` (32-bit).
Int64 literals such as `9223372036854775807` are stored by truncating to the
lower 32 bits, yielding `-1`.  The range-check comparison in codegen then uses
whatever integer width the self-hosted compiler assigns to the comparison
operands — also 32-bit — so the check cannot distinguish a large positive
value from a small negative one.

*Implementation path.*

. Add `ConstValueInt64: Int64` alongside `ConstValue: Integer` in `TSymbol`;
  populate it in `RegisterBuiltIns` and the constant-declaration semantic pass.
. In `EmitIntLiteralExpr` (and the constant-emit path), use `ConstValueInt64`
  when the resolved type is `tyInt64`; emit `l copy <value>`.
. Extend the range-check guard to compare `ConstValueInt64 > 2147483647`
  using a 64-bit comparison.
. Once the fix is verified in the self-hosted binary, `MaxInt` may optionally
  be changed back to `Int64 = 9223372036854775807`, though the 32-bit value
  is semantically correct for all current RTL uses.

*Effort.* Small — field addition and emit-path branch.

=== Single-precision intrinsic dispatch for trig functions

*What.* When a `Single` (32-bit float) operand is passed to a standard trig
function (`Sin`, `Cos`, `Tan`, `ArcTan`, `Sinh`, `Cosh`, `Tanh`, and their
inverses), the codegen should call the corresponding `*f` variant from libc
(`sinf`, `cosf`, `tanf`, `atanf`, `sinhf`, `coshf`, `tanhf`, etc.) rather
than the double-precision variant (`sin`, `cos`, etc.).

*Why it matters.* The x87 FPU always computed in 80-bit extended precision
regardless of operand type, so no speed difference between `Single` and
`Double` existed there. On modern x86_64, SSE2 scalar instructions
(`ADDSS`, `MULSS`, `SQRTSS`) do operate at genuine 32-bit precision, but
trig is computed via software polynomial approximations in libc — and that
is where the gap lies. The `*f` variants require a polynomial with roughly
half as many terms (24 mantissa bits vs 53), yielding a 2–4× speedup in
scalar trig compared to the `double` path. FPC and Delphi both call the
`double` variants for `Single` arguments, converting `Single → Double →
call sin() → Single` and paying the full double-precision cost plus two
conversions. This makes `Single` trig no faster than `Double` trig in
those compilers — a known pain point for real-time audio and signal
processing code.

*Implementation notes.*

* QBE already distinguishes `s` (single) and `d` (double) types, so the
  operand type is available at emit time.
* The fix is a small dispatch in the trig intrinsic call emitter: check
  the resolved type of the argument; if `tyFloat` (Single), emit a call to
  the `*f` symbol; if `tyDouble`, emit a call to the unprefixed symbol.
* No ABI or calling-convention changes are required; both families use
  the standard System V AMD64 float argument registers.
* A future extension could call a SIMD math library (e.g. SLEEF) for
  vectorised trig across arrays of singles, but that is a separate and
  larger undertaking.

*Relevant use cases.* Real-time PCM audio synthesis, DSP filters, FFT
pre/post-processing, simple video effects pipelines — all domains where
single precision is sufficient and throughput matters.

== Language — Features Required to Complete the punit Port

The following features were identified as missing from Blaise while porting
`/data/devel/fpc-3.3.1/src/packages/fcl-base/tests/punit.pp` to
`rtl/src/test/pascal/punit.pas`.  Each one has a concrete workaround in the
current port, noted below, so none is a blocker; they are collected here so
they can be prioritised alongside other language work.

=== Dynamic arrays with `SetLength`

*What.* Support local variables of type `array of T` whose length can be set
at runtime via `SetLength(A, N)`, with element access via `A[I]`.

*Why it matters.* `SetLength` is the idiomatic Pascal way to build variadic
collections.  In punit it is used for the `TSuiteArray` hook argument passed
to `TRunStartHandler` — a list of suites being run, built and freed per run.
Any future Blaise code that wants a heap-backed list without pulling in the
full `TList<T>` generic will hit the same gap.

*Current workaround in punit.* `TSuiteArray` was replaced with `PSuiteArray =
Pointer` (a raw `GetMem` block).  `TRunStartHandler` signature changed from
`procedure(const Suites: TSuiteArray)` to `procedure(Count: Integer; Suites:
PSuiteArray)`.  Helper functions `BlockGet` / `BlockSet` access pointer slots
by pointer arithmetic.  This works but is unidiomatic and requires manual
memory management at every call site.

*Implementation notes.* Dynamic arrays need: a heap-backed header (length +
capacity + ARC refcount), `SetLength` as a compiler built-in that calls into
the RTL to realloc the backing store, and bounds-checked index emission in
the codegen.  The ARC model already in place for strings is the natural
template.

*Effort.* Significant — new type kind, RTL support, codegen changes.

=== `Str()` intrinsic — numeric to string conversion

*What.* The FPC/Pascal `Str(X, S)` intrinsic writes the string representation
of a numeric value `X` (integer or float) into the `var` string parameter `S`.

*Why it matters.* It is the traditional Pascal way to convert a number to a
string without pulling in `SysUtils`.  Used throughout punit's `SysTimeString`
helper and the float assertion overloads.

*Current workaround in punit.* Integer conversions replaced with `IntToStr()`
and `Int64ToStr()`.  Float→string conversions use `DoubleToStr()` /
`SingleToStr()`.  The FPC-style `Str(X, S)` var-parameter form is still absent.

*Implementation notes.* `Str` is a thin wrapper: for integers, alias
`IntToStr` / `Int64ToStr`; for floats, alias `DoubleToStr` / `SingleToStr`.
The `var`-parameter assignment can be emitted as a string copy into `S`.

*Effort.* Trivial.

=== `New()` and `Dispose()` built-ins for typed pointers

*What.* `New(P)` allocates `SizeOf(P^)` bytes and returns a typed pointer;
`Dispose(P)` frees it.  Both are traditional Pascal alternatives to manual
`GetMem` / `FreeMem` with explicit `SizeOf`.

*Why it matters.* In the original punit every record allocation uses `New` and
every deallocation uses `Dispose`.  Without them the port must call
`GetMem(SizeOf(TFoo))` and `ZeroMem` manually, and the developer must remember
to `FreeMem` at every exit path.

*Current workaround in punit.* All uses replaced with `GetMem(SizeOf(T))` +
`ZeroMem(ptr, SizeOf(T))` + `FreeMem(ptr)`.

*Implementation notes.* `New(P)` is a thin wrapper over `GetMem(SizeOf(P^))`
with the type deduced from the pointer's declared type at the call site.
`Dispose(P)` calls `FreeMem(P)` after invoking the destructor if the pointed-to
type is a class (for record pointers, no destructor call is needed under ARC
provided string fields have been properly finalised first).

*Effort.* Small — semantic pass extracts element type, codegen emits the
appropriate `malloc` / `free` sequence.

=== `Default(T)` expression

*What.* `Default(T)` returns the zero value for any type `T` — zero for
integers, empty string for strings, nil for pointers and classes, and so on.
Commonly used to reset a record variable to its initial state.

*Why it matters.* In the original punit it is used to re-initialise
`TResultRecord` and `TRunSummary` records before each test run.

*Current workaround in punit.* Replaced with `ZeroMem(@rec, SizeOf(rec))`.
This is correct for records whose managed fields (strings, interface refs) are
already nil, but for a record mid-lifetime it bypasses ARC cleanup of any
non-nil string fields.  For punit's specific use (zeroing a freshly allocated
or just-finalised record) this is safe; for general use it is not.

*Implementation notes.* The semantic pass resolves the type argument; the
codegen emits a `ZeroMem` (or a field-by-field assignment for managed types
where ARC release must precede zeroing).

*Effort.* Small for unmanaged types; moderate for records containing managed
fields (requires field-by-field ARC release before memset).

=== `Finalize()` procedure for managed records

*What.* `Finalize(V)` releases all ARC-managed fields inside a record variable
`V` (strings, interface references, object references) without freeing the
record's own memory.  Used before `FreeMem` when the record was heap-allocated.

*Why it matters.* In the original punit it is called on `TResultRecord^` before
`FreeMem` to release the `TestMessage: string` field.  Without it, heap-
allocated records that contain strings leak the string's reference count.

*Current workaround in punit.* The `Finalize` calls were dropped.  In Blaise,
ARC on assignment should automatically manage string fields inside records, so
the string is released when the record's string field is overwritten or when the
last reference to the containing heap block is released.  The correctness of
this assumption depends on the ARC implementation for record fields — if the
codegen does not insert `_StringRelease` when a record pointer is freed via
`FreeMem`, there will be a string leak.

*Implementation notes.* `Finalize(V)` is a compiler built-in that walks the
fields of the record type and emits the appropriate ARC release calls for each
managed field.  The implementation mirrors what the codegen already does for
local record variables at scope exit.

*Effort.* Small to moderate — shares infrastructure with ARC scope-exit emit.

== BCL Packages — Numeric Types

=== `Currency` and `BigDecimal` as BCL packages

*What.* Two numeric types useful in financial and high-precision contexts that
were deliberately omitted from the Blaise primitive type set:

* `Currency` — a fixed-point decimal type represented as a 64-bit integer
  scaled by 10,000.  Exact decimal arithmetic with four decimal places;
  fast (integer operations), bounded range (~±922 trillion).  Equivalent to
  Delphi's `Currency` type.

* `BigDecimal` — arbitrary-precision decimal arithmetic, heap-allocated.
  Exact representation of any decimal value; no rounding error.  Equivalent
  to Java's `java.math.BigDecimal`.

*Why deferred.* Both are library-level concerns, not compiler primitives.  The
string-type consolidation precedent (all string variants → a single `string`)
applies equally here: the proliferation of `Currency`, `Comp`, `Extended`, and
`Real` came from hardware limitations and legacy concerns that no longer apply.
Blaise's two float primitives (`Double` and `Single`) cover the vast majority
of numeric use cases.

*Where they belong.* Once the BCL (`packages/`) exists:

* `bcl.numerics.fixed` — `TFixedDecimal` (Currency semantics, operator
  overloading, 4 decimal places).
* `bcl.numerics.bigdecimal` — `TBigDecimal` (arbitrary precision, operator
  overloading).

With operator overloading, both types will look like builtins from user code:

[source,pascal]
----
uses bcl.numerics.fixed;
var Price, Tax: TFixedDecimal;
begin
  Price := TFixedDecimal.FromStr('19.99');
  Tax   := Price * TFixedDecimal.FromStr('0.20');
  WriteLn(Tax.ToString);   { '4.0000' }
end.
----

*Migration path from FPC.* The migration analyser (future phase) will flag
`Currency` and `Comp` types and suggest `TFixedDecimal` or `TBigDecimal` as
replacements.  `Extended` maps to `Double` with a precision-loss warning.

*Effort.* Medium — primarily RTL/BCL work once operator overloading is in place.
No compiler changes needed.

== Tooling

=== Native Blaise test framework (fptest port)

*What.* Port the fptest framework to pure Blaise so that the compiler's own
test suite can be compiled and executed by the Blaise compiler, with no FPC
dependency in the test cycle.

*Why it matters.* The only remaining hard FPC dependency in the development
cycle is the test runner: `compiler/src/test/pascal/TestRunner.pas` and every
`cp.test.*.pas` unit import `fptest` units (`TestFramework`, `TextTestRunner`)
that are compiled by FPC.  `pasbuild test` therefore always requires FPC even
after the production compiler and RTL are fully self-hosting.  A native Blaise
port eliminates that dependency and closes the loop on a fully FPC-free
development cycle.

*Why fptest specifically.* fptest is an existing framework authored by the
same team, based on the original DUnit2 design.  It is faster than FPCUnit,
supports parameterised tests, and has a clean API.  Its source lives at
`/data/devel/fptest/`.  Porting it to Blaise rather than designing a new
framework preserves the existing test API (`TTestCase`, `CheckEquals`,
`CheckTrue`, `RegisterTest`, etc.) so that the existing `cp.test.*.pas`
units require only minor adaptation.

*Scope of the Blaise port.*

* Core classes: `TTestCase`, `TTestSuite`, `TTestResult`.
* Assert helpers: `CheckTrue`, `CheckFalse`, `CheckEquals`, `CheckNotEquals`,
  `AssertTrue`, `AssertFalse`, `Fail`.
* A plain text runner that prints `OK (N tests)` or a failure list to
  stdout and exits with a non-zero code on failure (pasbuild `test` command
  reads the exit code).
* No GUI runner, no XML output — keep the port minimal for now.

*Migration path.*

. Write the Blaise port of fptest as a new RTL unit (e.g. `TestFramework.pas`
  in `rtl/src/main/pascal/`).
. Adapt `TestRunner.pas` to import the Blaise version instead of the FPC one.
. Verify all existing `cp.test.*.pas` units compile and pass under Blaise.
. Update pasbuild's `test` command to invoke the Blaise binary directly instead
  of an FPC-compiled test runner.

*Effort.* Moderate.  fptest's core is small; the main effort is ensuring the
Blaise class system supports the TTestCase inheritance hierarchy and the
virtual dispatch pattern fptest relies on.

=== PasBuild: rename `--fpc` to `--compiler`

*What.* Rename PasBuild's `--fpc <path>` flag to `--compiler <path>`, keeping
`--fpc` as a deprecated alias.  Update help text and documentation accordingly.

*Why it matters.* The Blaise compiler already accepts FPC-style invocations:
it responds to `-iV`/`-iTP`/`-iTO` probes, maps `-Fu`/`-FE` to its internal
unit-path and output-dir, and silently ignores `-Mobjfpc`, `-O<n>`, and other
FPC flags.  So `pasbuild compile --fpc releases/v0.3.0/blaise` already works
today — the only friction is the misleading flag name.  Renaming it to
`--compiler` makes the intent clear and removes the implication that only FPC
binaries are accepted.

*Implementation notes.* Change is confined to `PasBuild.CLI.pas`
(flag parsing, help text) and `PasBuild.Utils.pas` (stored field name).
One-line alias: if `--fpc` is seen, treat it identically to `--compiler`.

*Effort.* Trivial.

== Stream I/O

=== Formal TStream decorator hierarchy

*What.* Define a `TStream` / `TFilterStream` base pair in the Blaise RTL,
together with canonical concrete implementations: `TFileStream`,
`TBufferedStream`, `TDataReader`, and `TDataWriter`.

*Why it matters.* FPC and Delphi define `TStream` as an abstract base, but
the standard library lacks a `TFilterStream` root for decorator chains.
Each unit that wraps a stream (ZLib, Cipher, LZMA) reinvents the
constructor convention, making chains non-uniform and difficult to compose.
A `TFilterStream` base ensures that every decorator follows the same
ownership and lifetime contract, enabling composable pipelines such as:

[source,pascal]
----
Pipe := TBufferedStream.Create(TFileStream.Create(Path, fmOpenRead));
----

*ARC advantage.* Under Blaise’s ARC model the outer stream holds a strong
reference to the inner stream; teardown is automatic when the outermost
wrapper goes out of scope.  No explicit free-order management is required,
which eliminates a common source of bugs in FPC code that builds nested
stream chains.

*Implementation notes.*

* `TStream` — abstract base: `Read`, `Write`, `Seek`, `Size`, `Position`.
* `TFilterStream` — abstract decorator base: holds `FInner: TStream`; all
  abstract methods default to forwarding to `FInner`.
* `TDataReader` / `TDataWriter` — typed I/O over an arbitrary `TStream`
  (mirrors Java’s `DataInputStream` / `DataOutputStream`).
* Connection to `TProcess`: process I/O output is currently accumulated as
  a `string`.  Once `TStream` exists, `TProcessOutput` can expose a
  `TStream` interface, enabling streaming reads without a full buffer.

*Effort.* Moderate.  Blocked on abstract method support in the class system.

== Debugger

=== PIE (ASLR) support in PDR

*What.* When a Blaise binary is compiled with `--debug-opdf`, the compiler
currently passes `-no-pie` to gcc to ensure OPDF `.quad symbol` entries
contain absolute link-time addresses.  The PDR debugger should instead read
the binary's actual load base at runtime and adjust all OPDF addresses by the
ASLR slide, allowing debug-opdf builds to remain PIE.

*Why it matters.* PIE (Position Independent Executable) is the default on all
modern Linux distributions and is required for ASLR protection.  Disabling it
in debug builds is a pragmatic workaround that is acceptable for local
development but is not appropriate for any scenario where the binary is
distributed or run in a production-adjacent environment.  Lifting the
restriction makes `--debug-opdf` safe to enable in CI and staging environments.

*Implementation notes.*

* On Linux, `/proc/<pid>/maps` lists each memory mapping with its base
  address and the backing file.  At debugger attach time, read this file,
  find the entry whose backing path matches the target binary, and record
  its base address as `LoadBase`.
* On FreeBSD, PIE and ASLR are supported by default from FreeBSD 12 onwards.
  FreeBSD's procfs is not mounted by default, so `/proc/<pid>/map` is not
  reliable.  The correct approach is `sysctl(KERN_PROC_VMMAP, pid)`, which
  returns the full VM map for a process; find the entry matching the binary
  path and extract its start address as `LoadBase`.
* On macOS, the equivalent is `task_info(TASK_DYLD_INFO)` or parsing the
  Mach-O load commands; `DYLD_IMAGE_BASE` is available via the dyld shared
  cache APIs.
* At symbol-resolution time on all platforms: `RuntimeAddr = OPDFAddr + LoadBase`.
* Once PIE-aware address resolution is in place, remove the `-no-pie` guard
  in `Blaise.pas` (`CompileToNative`).

*Effort.* Small on Linux and FreeBSD (one sysctl/procfs read at attach time);
moderate on macOS (dyld integration).

== macOS — Debugging and Code-Signing

=== Debugger entitlement requirements

*What.* On macOS, a debugger binary must carry the
`com.apple.security.get-task-allow` entitlement (and, post-SIP, the binary
it attaches to must carry the same entitlement) before the kernel permits
`task_for_pid`.  The Blaise debugger will need a documented signing workflow
for macOS targets.

*Why it matters.* Without the entitlement, `ptrace` and Mach exception
ports are blocked by the kernel even for self-compiled binaries.  Open-source
users who cannot obtain an Apple Developer certificate ($99/year) need a
supported path.

*Recommended approach.*

. *Ad-hoc signing* — the user self-signs with a locally generated certificate.
  Works for development; the compiler should emit a post-build `codesign`
  invocation for debug configurations.

. *Debug-server architecture* — ship a small, separately signed stub
  (`blaisedbgserver`) that handles `task_for_pid` and communicates over a
  local socket.  The main debugger frontend requires no special entitlement.
  This mirrors LLDB’s `debugserver` design and avoids per-binary signing of
  every compiled Pascal program.

. *SIP caveat* — SIP blocks attachment to Apple-signed system processes.
  For user-compiled Pascal binaries, SIP is not an obstacle provided the
  binary carries `get-task-allow`.

*Debug-format note.* The Blaise compiler’s own debug information format
(OPDF or equivalent) avoids dependency on Mach-O `dSYM` bundles and
Apple’s `dsymutil`.  This is the correct design: DWARF/dSYM integration adds
fragile toolchain coupling without benefit for a self-hosted debugger.

*Effort.* Small (documentation + build-system integration for `codesign`);
moderate for the debug-server architecture.

== Concurrency

=== Threading and modern concurrency models

*What.* Blaise needs a concurrency story.  The plan is a layered approach:
a compatibility layer for existing Pascal code, and a modern concurrency
model that makes Blaise worth choosing over FPC/Delphi for concurrent
workloads.

*Layer 1 — TThread (Delphi/FPC compatibility).*

Thin wrapper around `pthread_create` (POSIX) or `CreateThread` (Win32).
Required for porting existing Pascal codebases.  Provides the familiar
`TThread.Create`, `Execute` override, `Synchronize`, `WaitFor`, and
`Terminate` / `Terminated` pattern.  Not inspiring, but necessary for
ecosystem compatibility.

*Layer 2 — Modern structured concurrency.*

The industry has converged on a few proven models:

* *Structured concurrency with async/await* (C#, Rust, JavaScript, Swift,
  Kotlin) — the compiler transforms `async` functions into state machines.
  Works well with I/O-bound code.  Requires language-level support in the
  parser and codegen but no complex runtime scheduler.

* *Virtual threads / green threads* (Java 21, Go goroutines, Erlang
  processes) — the runtime multiplexes many lightweight threads onto a
  small pool of OS threads.  User code looks synchronous.  Requires a
  scheduler in the RTL, which is a larger investment.

* *Structured task groups* (Swift `TaskGroup`, Java `StructuredTaskScope`)
  — tasks have parent-child relationships with automatic cancellation
  propagation and scope-bound lifetimes.  Prevents leaked tasks by
  construction.

The recommended path for Blaise:

. Start with `TThread` for compatibility.
. Add `async`/`await` as the primary modern concurrency primitive.  This
  is the most compiler-friendly approach — it is a source-to-source
  transform during semantic analysis, turning async functions into state
  machine classes.  It does not require a complex runtime scheduler.
. Layer structured task groups on top of async/await for fork-join
  parallelism with automatic cancellation.
. Consider virtual threads as a long-term option if the runtime grows to
  support a user-mode scheduler (significant investment).

*Why not just TThread.* `TThread` maps 1:1 to OS threads, which are
expensive to create (~1 MB stack each) and limited in number (~thousands).
Modern server and UI workloads routinely need tens of thousands of
concurrent tasks (network connections, background jobs, UI animations).
Lightweight async tasks are the standard solution across every modern
language ecosystem.

*Design considerations.*

* ARC interaction — async tasks capture variables by reference; the ARC
  system must ensure captured objects outlive the task.  Structured
  concurrency helps here because task lifetimes are lexically scoped.
* QBE constraints — QBE has no coroutine or `setjmp`-based context
  switching primitives.  `async`/`await` via state-machine transform
  does not require these — the compiler generates ordinary function calls
  between states.  Green threads would require platform-specific context
  switching (`swapcontext` or hand-rolled assembly), which is harder to
  express through QBE.
* Exception propagation — async tasks need a defined exception model.
  Structured concurrency naturally provides this: exceptions propagate
  to the parent scope and cancel sibling tasks.

*Effort.* Large.  TThread is small (1–2 weeks); async/await is a
significant compiler feature (parser, semantic, codegen transforms) that
warrants its own design document and phased implementation.  This is
post-v0.6.0 work.
