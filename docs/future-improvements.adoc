= Blaise Compiler — Future Improvements
:toc:
:toc-placement: preamble
:sectnums:

This document records optimisation opportunities, quality-of-life improvements,
and language enhancements that are worth pursuing but are deliberately deferred
from the current implementation phase. Each entry states what the improvement is,
why it matters, and any known implementation notes.

Items here are not committed to any release milestone. They exist so that good
ideas are not lost and so that the rationale is available when the time comes to
prioritise them.

== Code Generation

=== Int64 literal range detection in self-hosted binary

*What.* When a constant or literal value exceeds the 32-bit signed range
(`> 2147483647`), the codegen should emit `l copy <value>` (64-bit) rather
than `w copy <value>` (32-bit).  Currently the range check in
`EmitIntLiteralExpr` is `Value > 2147483647`, which uses a 32-bit comparison
in the self-hosted binary; an Int64 value whose lower 32 bits appear negative
is incorrectly classified as in-range.

*Why it matters.* The current workaround is that `MaxInt` is declared as a
32-bit `Integer` constant (`2147483647`) rather than a true 64-bit value.
This is correct for `Copy(S, N, MaxInt)` usage (any string shorter than 2 GB
qualifies), but it means `High(Int64)` cannot be represented as a named
constant, and large Int64 literals in user code may be silently truncated.

*Root cause.* `TSymbol.ConstValue` is declared as `Integer` (32-bit).
Int64 literals such as `9223372036854775807` are stored by truncating to the
lower 32 bits, yielding `-1`.  The range-check comparison in codegen then uses
whatever integer width the self-hosted compiler assigns to the comparison
operands — also 32-bit — so the check cannot distinguish a large positive
value from a small negative one.

*Implementation path.*

. Add `ConstValueInt64: Int64` alongside `ConstValue: Integer` in `TSymbol`;
  populate it in `RegisterBuiltIns` and the constant-declaration semantic pass.
. In `EmitIntLiteralExpr` (and the constant-emit path), use `ConstValueInt64`
  when the resolved type is `tyInt64`; emit `l copy <value>`.
. Extend the range-check guard to compare `ConstValueInt64 > 2147483647`
  using a 64-bit comparison.
. Once the fix is verified in the self-hosted binary, `MaxInt` may optionally
  be changed back to `Int64 = 9223372036854775807`, though the 32-bit value
  is semantically correct for all current RTL uses.

*Effort.* Small — field addition and emit-path branch.

=== Single-precision intrinsic dispatch for trig functions

*What.* When a `Single` (32-bit float) operand is passed to a standard trig
function (`Sin`, `Cos`, `Tan`, `ArcTan`, `Sinh`, `Cosh`, `Tanh`, and their
inverses), the codegen should call the corresponding `*f` variant from libc
(`sinf`, `cosf`, `tanf`, `atanf`, `sinhf`, `coshf`, `tanhf`, etc.) rather
than the double-precision variant (`sin`, `cos`, etc.).

*Why it matters.* The x87 FPU always computed in 80-bit extended precision
regardless of operand type, so no speed difference between `Single` and
`Double` existed there. On modern x86_64, SSE2 scalar instructions
(`ADDSS`, `MULSS`, `SQRTSS`) do operate at genuine 32-bit precision, but
trig is computed via software polynomial approximations in libc — and that
is where the gap lies. The `*f` variants require a polynomial with roughly
half as many terms (24 mantissa bits vs 53), yielding a 2–4× speedup in
scalar trig compared to the `double` path. FPC and Delphi both call the
`double` variants for `Single` arguments, converting `Single → Double →
call sin() → Single` and paying the full double-precision cost plus two
conversions. This makes `Single` trig no faster than `Double` trig in
those compilers — a known pain point for real-time audio and signal
processing code.

*Implementation notes.*

* QBE already distinguishes `s` (single) and `d` (double) types, so the
  operand type is available at emit time.
* The fix is a small dispatch in the trig intrinsic call emitter: check
  the resolved type of the argument; if `tyFloat` (Single), emit a call to
  the `*f` symbol; if `tyDouble`, emit a call to the unprefixed symbol.
* No ABI or calling-convention changes are required; both families use
  the standard System V AMD64 float argument registers.
* A future extension could call a SIMD math library (e.g. SLEEF) for
  vectorised trig across arrays of singles, but that is a separate and
  larger undertaking.

*Relevant use cases.* Real-time PCM audio synthesis, DSP filters, FFT
pre/post-processing, simple video effects pipelines — all domains where
single precision is sufficient and throughput matters.

== Language

=== `const` sections in unit interface and implementation blocks

*What.* Allow `const` declaration sections to appear inside both the
`interface` and `implementation` blocks of a unit, not only at program
scope.  A unit's interface `const` section exports named constants to all
importing units.

*Why it matters.* Without this, RTL units that want to publish symbolic
constants must either (a) declare them at implementation scope (invisible to
callers) or (b) inline integer literals at every call site.  The current
workaround in `classes.pas` is to keep `dupAccept = 0`, `dupIgnore = 1`, and
`dupError = 2` in the interface section's `const` block — but the unit
parser does not yet parse that block, so the constants are absent from the
RTL public API and importing units must repeat the literals.

*Implementation notes.*

* `uParser.pas`: `ParseUnit` already dispatches on `tkConst` in the
  interface section to skip to the next keyword.  Change that to call
  `ParseConstSection` and attach the resulting constants to the unit scope.
* `uSemantic.pas` / `uSymbolTable.pas`: when a unit's interface section is
  imported, its constants must be registered in the importing scope under
  the unit namespace, consistent with how exported types and procedures are
  handled.
* No codegen change is required: constants are resolved at semantic time and
  folded to literals before IR emission.

*Effort.* Small.  Parser change is the critical path; symbol-table import
propagation follows the existing unit-import pattern.

=== Multi-line string literals

*What.* Add a raw-string or heredoc literal syntax to Blaise so that
multi-line text — especially inline source snippets used in compiler tests
— can be written without explicit `LineEnding` concatenation.

*Why it matters.* The compiler test suite contains many constants of the
form:

[source,pascal]
----
SrcFoo =
  'program P;'       + LineEnding +
  'var X: Integer;'  + LineEnding +
  'begin'            + LineEnding +
  '  X := 1'        + LineEnding +
  'end.';
----

The concatenation noise makes the inline source hard to read, and
mismatched `LineEnding` vs `#10` introduces subtle platform inconsistencies.
A raw-string literal would reduce these test constants to readable inline
Pascal source and eliminate the concatenation entirely.

*Candidate syntaxes.*

==== Option A: Heredoc (`<<'SENTINEL' ... SENTINEL`)

Borrowed from shell and Ruby.  The opening token `<<'EOF'` signals a
heredoc; the lexer consumes all lines verbatim until it finds a line
containing only the sentinel (here `EOF`).  Single quotes in the delimiter
suppress interpolation (matching `$'...'` shell convention).

[source,pascal]
----
{ Before }
SrcFoo =
  'program P;'       + LineEnding +
  'var X: Integer;'  + LineEnding +
  'begin'            + LineEnding +
  '  X := 1'        + LineEnding +
  'end.';

{ After — heredoc }
SrcFoo = <<'END_SRC'
program P;
var X: Integer;
begin
  X := 1
end.
END_SRC;
----

The sentinel must appear on a line by itself (leading/trailing whitespace
stripped).  Embedded single quotes and percent signs require no escaping.
The trailing newline before the sentinel is included in the string.

==== Option B: Raw string (backtick `` `...` ``)

Shorter syntax for small literals; the backtick is not currently used by
Pascal and is available as a delimiter.  No escape processing occurs inside
the backtick pair; a literal backtick cannot appear in the string.

[source,pascal]
----
{ Before }
SrcFoo =
  'program P;'       + LineEnding +
  'var X: Integer;'  + LineEnding +
  'begin'            + LineEnding +
  '  X := 1'        + LineEnding +
  'end.';

{ After — backtick raw string }
SrcFoo = `program P;
var X: Integer;
begin
  X := 1
end.`;
----

The inability to embed a backtick is a meaningful limitation for Pascal
source literals that contain character literals (e.g. test inputs that use
`#10`), but the compiler test suite does not currently use backticks in any
inline source.

==== Option C: Curly-brace raw string (`{{{ ... }}}`)

Inspired by Groovy and Kotlin.  Triple curly braces are unambiguous (Blaise
uses `{ }` for comments but not `{{{ }}}`).

[source,pascal]
----
{ Before }
SrcFoo =
  'program P;'       + LineEnding +
  'var X: Integer;'  + LineEnding +
  'begin'            + LineEnding +
  '  X := 1'        + LineEnding +
  'end.';

{ After — triple-brace raw string }
SrcFoo = {{{
program P;
var X: Integer;
begin
  X := 1
end.
}}};
----

Visually heavier than the backtick form; the extra braces make the start
and end of the literal unambiguous in a multi-column diff.


==== Option D: Triple single-quote (`'''...''' `)

Pascal is a single-quote language; triple single-quotes follow naturally
from the existing `''` escape convention and mirror Java's triple-quote
string blocks (Java 13+).  A single quote inside the block does not need
to be escaped.  The compiler strips the common leading whitespace margin
using the column position of the closing `'''`.

[source,pascal]
----
{ Before }
SrcFoo =
  'program P;'       + LineEnding +
  'var X: Integer;'  + LineEnding +
  'begin'            + LineEnding +
  '  X := 1'        + LineEnding +
  'end.';

{ After — triple single-quote }
SrcFoo = '''
  program P;
  var X: Integer;
  begin
    X := 1
  end.
  ''';
----

The closing `'''` determines the indentation baseline; all lines are
de-dented by that many columns.  Embedded single quotes (e.g. `'John'`)
require no escaping within the block.

*Disambiguation note.* The sequence `''''` (four single quotes) is valid
Pascal today and means two empty strings concatenated.  With triple-quote
support, the tokeniser must choose between reading `'''` as the opening of
a triple-quote block or reading the first `''` as an empty string followed by
`'` starting a new literal.  A two-token lookahead resolves the ambiguity:
if `'''` is followed by a newline or a printable character rather than a
closing `'''`, it opens a block.

==== Option E: Keyword heredoc (`beginstring SENTINEL ... SENTINEL`)

A keyword-based approach using `beginstring` with a user-supplied sentinel:

[source,pascal]
----
{ Before }
SrcFoo =
  'program P;'       + LineEnding +
  'var X: Integer;'  + LineEnding +
  'begin'            + LineEnding +
  '  X := 1'        + LineEnding +
  'end.';

{ After — keyword heredoc }
SrcFoo = beginstring ENDSRC
program P;
var X: Integer;
begin
  X := 1
end.
ENDSRC;
----

Zero delimiter collision: since `ENDSRC` is chosen by the programmer, it
can never appear accidentally inside the string content.  Slightly more
verbose than the heredoc or backtick forms, but eliminates all ambiguity
for large embedded source blocks.

==== Recommendation

The heredoc form (Option A) is preferred.

* Compatible with the existing single-pass lexer — scan until the closing
  sentinel appears alone on a line; no multi-character lookahead needed.
* Embedded single quotes require no escaping — essential for Pascal source
  literals in the test suite.
* Widely familiar from shell, Ruby, and SQL.

Option D (triple single-quote) is the strongest alternative: it is the
most visually natural choice for a Pascal programmer and directly analogous
to Java text blocks.  Its main complexity is the disambiguation rule for
`''''` sequences, which requires documented precedence.

*Effort.* Moderate.  Lexer state machine to collect heredoc/raw-string
content; parser treats the result as a `TStringLit` node; no semantic or
codegen changes beyond what regular string literals already require.



== Stream I/O

=== Formal TStream decorator hierarchy

*What.* Define a `TStream` / `TFilterStream` base pair in the Blaise RTL,
together with canonical concrete implementations: `TFileStream`,
`TBufferedStream`, `TDataReader`, and `TDataWriter`.

*Why it matters.* FPC and Delphi define `TStream` as an abstract base, but
the standard library lacks a `TFilterStream` root for decorator chains.
Each unit that wraps a stream (ZLib, Cipher, LZMA) reinvents the
constructor convention, making chains non-uniform and difficult to compose.
A `TFilterStream` base ensures that every decorator follows the same
ownership and lifetime contract, enabling composable pipelines such as:

[source,pascal]
----
Pipe := TBufferedStream.Create(TFileStream.Create(Path, fmOpenRead));
----

*ARC advantage.* Under Blaise’s ARC model the outer stream holds a strong
reference to the inner stream; teardown is automatic when the outermost
wrapper goes out of scope.  No explicit free-order management is required,
which eliminates a common source of bugs in FPC code that builds nested
stream chains.

*Implementation notes.*

* `TStream` — abstract base: `Read`, `Write`, `Seek`, `Size`, `Position`.
* `TFilterStream` — abstract decorator base: holds `FInner: TStream`; all
  abstract methods default to forwarding to `FInner`.
* `TDataReader` / `TDataWriter` — typed I/O over an arbitrary `TStream`
  (mirrors Java’s `DataInputStream` / `DataOutputStream`).
* Connection to `TProcess`: process I/O output is currently accumulated as
  a `string`.  Once `TStream` exists, `TProcessOutput` can expose a
  `TStream` interface, enabling streaming reads without a full buffer.

*Effort.* Moderate.  Blocked on abstract method support in the class system.

== macOS — Debugging and Code-Signing

=== Debugger entitlement requirements

*What.* On macOS, a debugger binary must carry the
`com.apple.security.get-task-allow` entitlement (and, post-SIP, the binary
it attaches to must carry the same entitlement) before the kernel permits
`task_for_pid`.  The Blaise debugger will need a documented signing workflow
for macOS targets.

*Why it matters.* Without the entitlement, `ptrace` and Mach exception
ports are blocked by the kernel even for self-compiled binaries.  Open-source
users who cannot obtain an Apple Developer certificate ($99/year) need a
supported path.

*Recommended approach.*

. *Ad-hoc signing* — the user self-signs with a locally generated certificate.
  Works for development; the compiler should emit a post-build `codesign`
  invocation for debug configurations.

. *Debug-server architecture* — ship a small, separately signed stub
  (`blaisedbgserver`) that handles `task_for_pid` and communicates over a
  local socket.  The main debugger frontend requires no special entitlement.
  This mirrors LLDB’s `debugserver` design and avoids per-binary signing of
  every compiled Pascal program.

. *SIP caveat* — SIP blocks attachment to Apple-signed system processes.
  For user-compiled Pascal binaries, SIP is not an obstacle provided the
  binary carries `get-task-allow`.

*Debug-format note.* The Blaise compiler’s own debug information format
(OPDF or equivalent) avoids dependency on Mach-O `dSYM` bundles and
Apple’s `dsymutil`.  This is the correct design: DWARF/dSYM integration adds
fragile toolchain coupling without benefit for a self-hosted debugger.

*Effort.* Small (documentation + build-system integration for `codesign`);
moderate for the debug-server architecture.
